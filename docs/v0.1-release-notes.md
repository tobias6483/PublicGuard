# PublicGuard v0.1.0 Release Notes Draft

PublicGuard v0.1.0 is an early unsigned macOS menu bar beta for local laptop
protection in public workspaces. It is intended for maintainer testing and
early technical feedback before signed distribution.

## Highlights

- Native Swift/AppKit menu bar app for macOS 14 and newer.
- Local Arm/Disarm flow protected by Touch ID or password.
- Charger disconnect, Wi-Fi change, sleep/wake, Bluetooth proximity, and idle
  timeout triggers.
- Loud alarm and silent response modes.
- Configurable grace periods, per-trigger grace overrides, and trigger
  cooldowns for false-positive tuning.
- Bundled local alarm sounds with normal and maximum app playback volume.
- Optional lock screen action.
- Optional launch at login for app bundle builds.
- Local-only event log with standard, minimal, and encrypted storage options.
- Event log retention controls with 7 day, 30 day, forever, and manual prune
  options.
- Recent Events and Recent Trigger Status menu diagnostics.
- Unsigned GitHub Actions app artifact with bundle validation and SHA-256
  checksum.

## Privacy And Security Notes

- No account.
- No analytics.
- No cloud sync.
- No hidden recording.
- No location tracking.
- Event logs stay on the Mac.
- Bluetooth proximity stores only the user-confirmed local device identifier and
  display name in user defaults. It is passive BLE scanning, not Bluetooth
  pairing or proof of ownership.
- Camera snapshots are not included in v0.1. Any future camera behavior must
  pass the review gate in [camera-snapshot-privacy-review.md](camera-snapshot-privacy-review.md).

## Known Limits

- The app artifact is unsigned and not notarized.
- macOS may warn before opening the app.
- Closed-lid alarm playback is not guaranteed while a MacBook is asleep.
- Sleep/wake behavior depends on macOS delivering lifecycle notifications before
  suspension and after wake.
- Lock screen behavior depends on macOS accepting either the legacy `CGSession`
  command when present or the `pmset displaysleepnow` fallback.
- Bluetooth proximity is experimental and depends on the target device's BLE
  advertising behavior and the user confirming the intended nearby candidate.
- CoreBluetooth proximity alone does not cryptographically prove phone
  ownership; stronger assurance would require a future verified companion or
  pairing flow, such as challenge/response.
- iPhones may not advertise a stable BLE identity in every state.
- The app has not yet gone through a signed Developer ID release pipeline.
- Homebrew distribution is not included.
