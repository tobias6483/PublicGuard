# PublicGuard v0.1.1 Release Notes Draft

PublicGuard v0.1.1 is an early unsigned macOS menu bar beta for local laptop
protection in public workspaces. It is intended for maintainer testing and
early technical feedback before signed distribution. This patch release updates
the public beta to the latest MVP hardening on `main` after `v0.1.0`.

## Highlights

- Native Swift/AppKit menu bar app for macOS 14 and newer.
- Local Arm/Disarm flow protected by Touch ID or password.
- Charger disconnect, Wi-Fi change, sleep/wake, Bluetooth proximity, and idle
  timeout triggers.
- Loud alarm and silent response modes.
- Configurable grace periods, per-trigger grace overrides, and trigger
  cooldowns for false-positive tuning.
- Bundled local alarm sounds with normal and maximum app playback volume.
- Optional lock screen action.
- Optional launch at login for app bundle builds.
- Local-only event log with standard, minimal, and encrypted storage options.
- Event log retention controls with 7 day, 30 day, forever, and manual prune
  options.
- Recent Events and Recent Trigger Status menu diagnostics.
- Unsigned GitHub Actions app artifact with bundle validation and SHA-256
  checksum.

## Changes Since v0.1.0

- Hardened authentication and grace-period behavior so failed authentication
  fails closed and pending grace periods can be cancelled safely.
- Preserved active alarm state when arming and when changing response modes or
  presets, so active alarms still require authenticated stop/disarm.
- Improved trigger latency and false-positive controls for Wi-Fi, charger
  disconnects, trigger cooldowns, and per-trigger grace overrides.
- Hardened Bluetooth proximity learning and baseline behavior with repeated
  observation checks before saving a learned device.
- Hardened event log formatting, encrypted append-only records, active-storage
  pruning, and retention behavior.
- Preserved an explicit zero-trigger configuration across relaunches.
- Hardened lock screen command handling so PublicGuard falls back or logs
  failure when a command exits unsuccessfully.
- Updated CI and artifact actions to current Node.js-compatible major versions.

## Privacy And Security Notes

- No account.
- No analytics.
- No cloud sync.
- No hidden recording.
- No location tracking.
- Event logs stay on the Mac.
- Bluetooth proximity stores only the user-confirmed local device identifier and
  display name in user defaults. It is passive BLE scanning, not Bluetooth
  pairing or proof of ownership.
- Camera snapshots are not included in v0.1. Any future camera behavior must
  pass the review gate in [camera-snapshot-privacy-review.md](camera-snapshot-privacy-review.md).

## Hardware QA

Hardware QA results are recorded in
[hardware-qa-results.md](hardware-qa-results.md). The automated release
baseline must pass on the release commit before publishing, but real-device
trigger and app-behavior QA has not yet been run.

Current release status:

- Tested: automated release baseline with `scripts/release_check.sh` on the
  release commit before tag publication.
- Partially tested: none recorded.
- Not tested: charger disconnect, Wi-Fi change/disconnect, Bluetooth proximity,
  idle timeout, sleep/wake, lid close/reopen, notifications, lock screen,
  launch at login, alarm sounds/volume, and event log storage menu UX in
  `dist/PublicGuard.app`.

These untested areas remain explicit for the v0.1.1 beta rather than implying
hardware coverage that has not happened yet.

## Known Limits

- The app artifact is unsigned and not notarized.
- macOS may warn before opening the app.
- Closed-lid alarm playback is not guaranteed while a MacBook is asleep.
- Sleep/wake behavior depends on macOS delivering lifecycle notifications before
  suspension and after wake.
- Lock screen behavior depends on macOS accepting either the legacy `CGSession`
  command when present or the `pmset displaysleepnow` fallback.
- Bluetooth proximity is experimental and depends on the target device's BLE
  advertising behavior and the user confirming the intended nearby candidate.
- CoreBluetooth proximity alone does not cryptographically prove phone
  ownership; stronger assurance would require a future verified companion or
  pairing flow, such as challenge/response.
- iPhones may not advertise a stable BLE identity in every state.
- The app has not yet gone through a signed Developer ID release pipeline.
- Homebrew distribution is not included.

## Download Notes

The v0.1 artifact, when published, should include:

- `PublicGuard.app.zip`
- `PublicGuard.app.zip.sha256`

The checksum verifies the downloaded ZIP bytes only. It is not a code-signing or
notarization substitute.

After verifying and unzipping the artifact, move `PublicGuard.app` to
`/Applications` for normal use. Running it directly from `Downloads` is fine for
a quick smoke test, but `/Applications` is the normal macOS app location, gives
launch-at-login and app identity behavior a more stable path, and keeps future
update instructions anchored to one location.
